# Worker liveness via heartbeat, and a single taxonomy for why research isn't progressing

Clicking **Run Research** can silently hang: the **Worker** (`pnpm worker`) may not be running, no runner CLI may be installed/logged-in, or the operator may have **paused** intake — and today all three are indistinguishable from "working" (the button at `components/product/research-button.tsx:50-71` polls `QUEUED` forever with no timeout). We make the Worker publish a **liveness heartbeat** and we define one operator-facing taxonomy — **paused / offline / no-runners / queued (busy) / in-flight** — surfaced consistently at every Run-Research trigger and on the Research page.

## Decision

- **Heartbeat is written by an independent `setInterval(5s).unref()` timer, never by the work loop.** The main loop (`server/worker/main.ts:45-54`) `await`s a whole job inside `processNextJob`, which can block for up to ~600s (3 parallel runners × up to two 300s attempts with the repair-retry). A heartbeat tied to loop iterations would therefore go stale *during every legitimate long run* and falsely report "Worker down" on the happy path. The timer fires regardless of what the loop is awaiting (the runners are I/O-bound child processes). Write the first heartbeat **synchronously at startup** (before the loop) and a `phase:"stopping"` write in the shutdown handler.
- **Storage:** the existing `kv` table, key `workerStatus` = `{ ts, pid, phase: "idle"|"busy"|"stopping", currentJobId, runners: { cursor:{installed,loggedIn}, codex:{…}, claude:{…} }, runnersCheckedAt }`. Confirmed clobber-safe: `writeState` (`server/store.ts:176-178`) only writes `settings`/`schemaOverrides`/`aggregatorOverrides`, and `kv` is an upsert keyed by `key`.
- **Liveness derivation:** `online` = `now - ts < STALE_MS` where `STALE_MS = 20000` (4× cadence — absorbs a missed tick / GC pause / short macOS nap); `down` = row present but stale; `unknown` = no row yet. The **UI debounces**: require two consecutive stale reads before showing "down" so a single post-sleep stale read doesn't flash a scary banner. `STALE_MS` is tied to the timer cadence, **not** to job duration.
- **One read, no racing endpoint:** enrich the existing `GET /api/research-jobs/[id]` poll response with `worker: { state, lastSeenMs, phase, currentJobId, runners }`; the button consumes that (it already polls every 1s). Add a separate `GET /api/research/worker-status` **only** for the idle Research-page banner (which has no job id to attach to).
- **Refuse doomed jobs at enqueue** (`POST /api/products/[id]/research-jobs`): `503` if Worker down/unknown, `422` if alive but zero runners installed. This converts "job created, silently fails 5 min later" into an immediate, correct message.
- **Paused intake is part of this taxonomy, not a separate feature.** The dead `fakeResearchMode` setting becomes a real **intake gate**: the authoritative check lives in `createResearchJob` (`server/store.ts:458`), placed **after** the existing dedupe branch, and returns a **discriminated result** (not an overloaded `null` — `null` already means "product not found" → 404). When paused the route returns `409 { code: "RESEARCH_PAUSED" }`, deliberately distinct from the `503` worker-down signal. Pause stops **new intake only**; jobs already `QUEUED`/in-flight drain (never gate `claimNextResearchJob`). The Run-Research triggers (`research-button.tsx`, `triage-dashboard.tsx`) render **disabled with an inline reason** when paused — fed by `getDemoSettings()` as a Server-Component prop — so paused is never confused with offline, and the operator never gets a post-click surprise.
- **Runner availability must report `loggedIn`, not just `installed`.** `getAvailableRunners` (`server/research-runner/index.ts:21-26`) currently filters on `installed` only, so an installed-but-logged-out CLI counts as available and fails at spawn time. The heartbeat snapshot and the banner surface both flags.

## Considered options

- **Pure client-side `QUEUED`-timeout heuristic** (no Worker change): "still `QUEUED` after ~6s ⇒ Worker not consuming the queue." Cheap and kills the *silent* hang, but cannot distinguish **down** from **busy on a prior job** (jobs run sequentially), cannot power the **idle** Research-page banner (no job to inspect), and cannot name the **missing runner**. Adopted as a secondary backstop; the heartbeat is primary.
- **Next server probes `getAvailableRunners()` on demand:** tells you about *runners*, not whether the *Worker loop* is alive, and spawns up to three 8s child processes per call — abusive on a 1s poll. Rejected for liveness; a single cheap probe at *enqueue* is fine.
- **A second polling endpoint for the button:** creates a TOCTOU race against the job poll (the two reads can disagree). Rejected in favour of enriching the existing poll.
- **Renaming `fakeResearchMode` → `researchEnabled`:** rejected for now. Settings persist as one JSON blob; `sanitizeSettings` falling back to the default `true` would **silently un-pause** an operator on upgrade, and it forces a `MOCK_DOMAIN_CONTRACT_VERSION` bump. Keep the key; the user-visible label ("Enabled/Paused") is already honest. If ever renamed, add `settings.researchEnabled ?? settings.fakeResearchMode` read-aliasing first.

## Consequences

The Worker now writes a tiny `kv` upsert every ~5s under WAL (negligible). The Settings "Research jobs: Enabled/Paused" toggle stops being a no-op. The dead `defaultResearchDelaySeconds` knob is removed (real timing is env-controlled via `RESEARCH_POLL_MS`/`RESEARCH_RUN_TIMEOUT_MS`); also remove the dead `getMockResearchAgentDurationMs`/`MOCK_RESEARCH_AGENT_SECONDS` once confirmed to have no consumer. A `~12min` hard client-side backstop stays on the poll loop as belt-and-suspenders.

**Goal:** clicking Run Research when the Worker is offline, no runners are available, or intake is paused always yields a specific, correct message within ~20s; the Research page shows Worker + per-runner status when idle; and no false "Worker down" ever appears during a legitimate multi-minute run.
